## 🚨 强制执行检查清单（每个任务开始前必读）

### 第一步：理解任务边界
- [ ] 我已经完整阅读了工单的所有内容
- [ ] 我明确知道需要修改哪些文件（精确的文件路径）
- [ ] 我明确知道需要创建哪些文件（精确的文件路径）
- [ ] 我明确知道不应该修改哪些文件

### 第二步：探索现有代码（必须执行）
在开始编码前，必须完成以下探索：

```bash
# 1. 检查目标文件是否存在
ls -la [目标文件路径]

# 2. 如果要修改现有文件，必须先读取完整内容
cat [目标文件路径]

# 3. 如果要添加路由链接，必须检查目标页面是否存在
ls -la app/[目标路由路径]/page.tsx
```

**❌ 禁止行为**：
- 不允许在未检查页面存在性的情况下，添加指向该页面的链接
- 不允许保留指向不存在页面的链接
- 不允许猜测文件路径或组件名称

### 第三步：验证依赖关系
- [ ] 我已经检查了所有需要导入的组件是否存在
- [ ] 我已经检查了所有需要调用的 API 接口是否存在
- [ ] 我已经检查了所有需要跳转的路由页面是否存在

### 第四步：执行编码
- [ ] 我的代码严格遵循工单中的实现细节
- [ ] 我没有添加工单中未要求的功能
- [ ] 我没有删除工单中未要求删除的代码

### 第五步：自我验证（提交前必须执行）
- [ ] 我已经运行 `npm run build` 确认没有编译错误
- [ ] 我已经检查了所有新增的路由链接指向的页面都存在
- [ ] 我已经检查了所有修改的文件语法正确
- [ ] 我已经测试了关键功能路径（如果工单中有说明）

## 📋 常见错误案例与避免方法

### ❌ 错误案例 1：保留指向不存在页面的链接

**错误示例**：
```typescript
// components/user-info.tsx
<DropdownMenuItem onClick={() => router.push('/profile')}>
  <span>个人资料</span>
</DropdownMenuItem>
```

**问题**：`/profile` 页面不存在，导致 404 错误

**正确做法**：
```bash
# 第一步：检查 /profile 页面是否存在
ls -la app/profile/page.tsx

# 第二步：如果不存在，检查是否有替代页面
ls -la app/account/page.tsx

# 第三步：使用存在的页面路径
```

```typescript
// 修改后
<DropdownMenuItem onClick={() => router.push('/account')}>
  <span>个人资料</span>
</DropdownMenuItem>
```

---

### ❌ 错误案例 2：对模糊描述的自主理解

**工单描述**：
> "从用户中心侧边栏可以进入"

**错误理解**：
- ❌ "用户中心侧边栏" = "顶部用户下拉菜单"
- ❌ 直接在 `components/user-info.tsx` 中添加菜单项

**正确做法**：
```bash
# 第一步：搜索项目中是否有"用户中心"相关的组件
find . -name "*account*" -o -name "*user-center*"

# 第二步：检查是否有侧边栏组件
find . -name "*sidebar*"

# 第三步：阅读现有的用户中心页面结构
cat app/account/layout.tsx
cat components/account/account-sidebar.tsx

# 第四步：根据实际代码结构决定实现方式
```

---

### ❌ 错误案例 3：路由命名的自主决策

**工单描述**：
> "路由地址：`/account/credits` 或 `/account/history`"

**错误做法**：
- ❌ 自己决定使用 `/account/transactions`（虽然更合理，但不符合工单）

**正确做法**：
```typescript
// 严格使用工单中的第一个选项
const route = '/account/credits'

// 或者，如果工单中有"推荐"标注，使用推荐的选项
```

**如果你认为工单中的命名不合理**：
1. 不要自作主张修改
2. 在执行报告中注明："建议将路由改为 `/account/transactions`，理由是..."
3. 等待 CTO 确认后再修改

## 🔧 标准执行流程模板

### 任务：修改现有组件

```bash
# 1. 读取目标文件
cat [文件路径]

# 2. 理解现有代码结构
# - 找到需要修改的具体位置
# - 确认修改不会破坏现有功能

# 3. 检查所有依赖
# - 如果添加新的导入，检查被导入的文件是否存在
# - 如果添加新的路由链接，检查目标页面是否存在

# 4. 执行修改
# - 使用精确的代码替换
# - 不要修改工单中未提到的部分

# 5. 验证修改
npm run build
```

### 任务：创建新页面

```bash
# 1. 检查目标路径是否已存在
ls -la app/[路由路径]/page.tsx

# 2. 如果已存在，停止执行，报告冲突
# 如果不存在，继续

# 3. 检查是否需要 layout 文件
ls -la app/[路由路径]/layout.tsx

# 4. 创建页面文件
# - 严格按照工单中的组件结构
# - 使用工单中指定的 UI 组件库

# 5. 验证创建
npm run build
```

## 🎯 核心原则总结

1. **先探索，后编码**
   - 不允许在未探索代码库的情况下开始编码
   - 必须验证所有依赖的存在性

2. **严格遵循工单**
   - 不允许自主决策架构层面的问题
   - 不允许自主修改路由命名、组件结构等

3. **完整验证**
   - 每次修改后必须运行 `npm run build`
   - 必须检查所有新增的链接是否有效

4. **及时报告**
   - 发现工单中的不明确之处，立即报告
   - 发现代码库中的问题，立即报告
   - 不要猜测，不要假设
